buildSendTx

  • buildSendTx(amount, recipient, tokenType, refKey?, memo?, extraFee?, isSolanaPayTransfer?, allowOwnerOffCurve?, customRecipientTA?, customFeeCollectorTA?, wardenInfo?): Promise<SendTxData>

  • Build a tx to send tokens to a specified recipient using the private balance

    Parameters

    • amount: number

      Amount of the token to be sent from the private balance. Must be at least the current price of rent for a token account.

    • recipient: PublicKey

      Recipient who should receive said amount of token. For the case of sending SPL-tokens, this is the recipient's owner account (i.e. the same one you'd provide when sending SOL) NOT a token account. Be aware of this, to avoid your tokens being sent to the wrong account. Internally this is then mapped to recipient's Associated Token Account.

    • tokenType: "LAMPORTS" | "USDC" | "USDT" | "mSOL" | "BONK" | "SAMO"

      Type of token to be sent

    • Optional refKey: PublicKey

      Optional reference key to be included in final tx of the send, useful for e.g. Solana Pay

    • Optional memo: string

      Optional memo to be included in final tx of the send, useful for e.g. Solana Pay. Maximum length 128 bytes in utf8.

    • extraFee: OptionalFee = ...

      Optional fee to be collected by a third party in the same TokenType as the send. If not provided, no fee will be collected. When specifying the collector here, make sure its ATA already exists. If you wish to use a custom TA for the collector, use the customFeeCollectorTA parameter.

    • isSolanaPayTransfer: boolean = ...

      Whether this is a transfer for Solana Pay. The main difference this makes is that the actual transfer of funds is done using a seperate Transfer instruction, as required by the Solana Pay spec. If it is set to false, the transfer is done using a CPI. Defaults to true if refKey is set, false otherwise.

    • allowOwnerOffCurve: boolean = false

      Controls wether we can send tokens to PDAs or other accounts whose public key does not lie on ed25519. Same idea as for the getAssociatedTokenAddressSync function from the spl library.

    • customRecipientTA: undefined | PublicKey = undefined

      For the case that you want to provide an override for a Token Account to send to that is not the recipient's ATA. Make sure this token account is funded with rent, else this will fail.

    • customFeeCollectorTA: undefined | PublicKey = undefined

      For the case that you want to provide an override for a Token Account to collect the OptionalFee that is not the collector in OptionalFee's ATA. Make sure this token account is funded with rent, else this will fail.

    • wardenInfo: WardenInfo = ...

      Controls what warden is used. Filled in with default config.

    Returns Promise<SendTxData>

buildTopUpTx

  • buildTopUpTx(amount, tokenType, manualMerge?, wardenInfo?, sender?): Promise<TopupTxData>

  • Build a tx to top up the private balance

    Parameters

    • amount: number

      Amount of the token to top up with

    • tokenType: "LAMPORTS" | "USDC" | "USDT" | "mSOL" | "BONK" | "SAMO"

      Type of token to topup with

    • manualMerge: boolean = false

      If too many topups (currently 4) are sent without a send inbetween (see docs on commitment management for more info), a merge is required. If this boolean is set to false, this merge is done before building the topup if needed (i.e. the sdk user does not need to worry about this). Note: This means, a topup might take longer to build at times as a merge needs to be built and confirmed prior. If this boolean is set to true, this method will throw until a merge has been performed manually by the sdk user.

    • wardenInfo: WardenInfo = ...

      Controls what warden is used. Filled in with default config.

    • sender: PublicKey = ...

      Optional sender to use for the topup. Defaults to the owner of the elusiv instance.

    Returns Promise<TopupTxData>

Static estimateTxsCountLastTime

  • estimateTxsCountLastTime(cluster, conn, timeSec, tokenType?, msBetweenBatches?): Promise<number>

  • Estimate the number of transactions that have been made in the last specified time frame via elusiv. This is a heuristic, and is not guaranteed to be accurate. The reason it exists is because getTxsCountLastTime is expensive in terms of rpc calls, and this is a significantly cheaper alternative.

    Parameters

    • cluster: Cluster

      Cluster on which to get the transaction count

    • conn: Connection

      Connection to use to make calls to solana blockchain

    • timeSec: number

      The number of seconds in the past to look at

    • Optional tokenType: "LAMPORTS" | "USDC" | "USDT" | "mSOL" | "BONK" | "SAMO"

      Type of token to fetch transaction count for

    • msBetweenBatches: number = 0

      The number of milliseconds to sleep between each batch of sigs fetched. This is to avoid spamming the RPC. Defaults to 0.

    Returns Promise<number>

    The estimated number of transactions (i.e. topups and sends, not normal solana txs) in the last specified time
